## 一、项目背景与核心需求

### 1.1 背景

- 专科学校有多个专业需要实操性训练，实训合格是学生毕业的前提
- 目前实训打分**全靠老师人工查看**，工作量巨大
- 目标：实现自动化打分，节省人力；学生可随时预约实训室，完成后由系统自动评分

### 1.2 合作模式

- 老师希望做成**长期合作**，细水长流
- 采用"**先交付成果，再申请预算**"模式，一期必须做出实际成果
- 二期可能涉及更复杂的视频处理需求
- **硬件由校方负责，我方只负责软件（前端 + 后端）**

## 二、一期系统设计

### 2.1 核心思路转变

> **关键认知：一期的系统输入不是视频，而是实训设备输出到数据库里的结构化数据。**

现有实训设备已具备逐步骤打分功能（例如步骤 1/2/3/4/5 的对/错判定），结果直接存入校方现有的 **MySQL 数据库**。我们的系统通过定时任务（Cron Job）拉取数据库数据进行分析处理。

### 2.2 系统输入与输出

| 项目 | 说明 |
|------|------|
| **输入** | 实训设备写入 MySQL 的结构化数据（每个步骤的对/错结果） |
| **输出** | 单次实训报告 + 长期能力分析报告 + 学生能力图谱 |
| **数据拉取方式** | Cron Job 每天定时从校方 MySQL 拉取最新实训结果 |

### 2.3 功能模块

#### 模块 1：实训结果分析

- 读取学生单次实训的步骤完成情况（每步对/错）
- 计算得分（如 5 步对了 3 步 = 60 分）
- 分析每个未完成步骤对应的能力欠缺
- 生成单次实训报告

#### 模块 2：能力模型与图谱

- **技能拆解**：学生毕业需掌握多种技能，每种技能可拆解为多个子技能
- **映射关系**：实训步骤 ↔ 子技能之间存在一对多或多对多关系
- **能力图谱**：汇总所有实训记录（最新 + 历史），将相近子能力归类为大类能力，生成五维或六维能力雷达图
- **毕业达标判定**：评估学生是否达到毕业标准，展示各项子能力的集齐程度

#### 模块 3：AI 报告生成

- AI 读取所有规则和映射关系，理解逻辑后为学生生成**用户友好的诊断报告**
- 以自然语言告诉学生：欠缺在哪里、应该如何提升
- 报告支持下载

#### 模块 4：实训室环境规范检查

- 学生完成实训后，对实训室现场拍摄照片并上传
- 大模型对比实训后的实训室照片与标准照片，判断：
  - 物品是否正确归位
  - 是否存在乱丢垃圾
  - 实训台是否清理干净
- 检测结果记录在报告中或存入数据库
- **待确认**：实训室照片如何获取（手动上传 vs 摄像头自动拍摄）；如涉及硬件（延时相机、定时拍摄、摄像头对接），需校方配合

#### 模块 5：用户系统

- 学生端：查看个人实训数据、报告、能力图谱
- 教师端：查看全部学生数据、班级汇总
- **待确认**：是否需要对接校园网统一认证系统（CAS/LDAP 等），还是独立账号体系

## 三、业务规则（待确认项）

### 3.1 计分与惩罚机制

需与老师商定具体规则，例如：
- 学生第一次实训第 4 步做错，第二次重做第 4 步对了但第 5 步又错了
- 最终得分如何计算？保留最高分？还是有惩罚机制？
- 本质上可通过**规则引擎**解决

### 3.2 映射关系的精确定义

- 实训步骤 → 子技能：一对多 or 多对多？
- 多个不同实训是否会共同影响同一个子技能的评分？
- 需要老师提供明确的映射表

## 四、项目规模

| 指标 | 数值 |
|------|------|
| 一期实训场景数 | 3 个（现有 3 个实训教室） |
| 学生总数 | 140 人 |
| 班级规模 | 约 40 人/班 |
| 实训频率 | 约每周一次 |
| 结果时效要求 | 学生接受次日出结果 |
| 未来扩展 | 年底前再建 4 个教室，后续再建 2 个，远期接近 10 种设备 |

**规模评估**：压力不大。大模型可顺序处理所有报告并存入数据库，无并发瓶颈。

## 五、技术挑战与风险

| 挑战 | 说明 | 应对 |
|------|------|------|
| 本地化部署 | 校方希望模型本地部署，不调用云端 | 开源模型能力有限，需评估可行性；老师理解此困难 |
| 实训室照片获取 | 实训室照片获取方式（摄像头 vs 手动上传）涉及硬件 | 硬件校方负责，但需确认对接方式 |
| 评分标准量化 | 部分标准可能存在主观性 | 通过规则引擎 + AI 辅助解决 |
| 数据库访问 | 需要能从公网访问校方 MySQL，或提供跳板机 | 需校方配合网络配置 |
